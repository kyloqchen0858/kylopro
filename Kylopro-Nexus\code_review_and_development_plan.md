# Kylopro-Nexus 代码审查与后续开发计划

基于对现有代码的初步审查（模拟 trae 检查结果），制定本计划以指导后续开发。

## 1. 代码质量概览

### 1.1 优点
- 模块化设计清晰，核心 provider、engine、skills 分离良好。
- 错误处理较为完备，关键操作有 try-ccept 包裹。
- 注释和文档较为齐全，SKILL.md 格式规范。
- 隐私保护考虑周到（文件内容本地处理，仅推送摘要）。

### 1.2 待改进项

#### 1.2.1 类型注解与静态检查
- 部分函数返回值类型未标注（如 `_read_file_preview` 返回 `str`，但未注明）。
- 某些变量类型为 `Any`，可进一步细化。
- 建议运行 `mypy` 或 `pyright` 进行静态类型检查，并逐步完善类型注解。

#### 1.2.2 错误处理与日志
- 部分异常捕获过于宽泛（`except Exception`），建议区分可恢复错误与不可恢复错误。
- 日志级别使用可优化：部分 debug 信息当前用 info 级别，可调整为 debug。
- 建议添加更详细的错误上下文（如文件名、行号、操作类型）便于排查。

#### 1.2.3 依赖管理
- `requirements.txt` 中版本未固定，可能导致环境不一致。
- 建议使用 `pip-tools` 或 `poetry` 管理依赖，并生成 `requirements-dev.txt` 包含开发工具。

#### 1.2.4 配置管理
- 目前配置分散在 `.env`、`~/.nanobot/config.json` 以及代码硬编码中。
- 建议统一配置加载机制，提供配置验证和默认值。

#### 1.2.5 测试覆盖
- 当前未见单元测试或集成测试。
- 应添加 `tests/` 目录，对核心模块（provider, engine, skills）编写单元测试。
- 考虑使用 `pytest` 和 `pytest-asyncio`。

## 2. 性能与可扩展性

### 2.1 异步处理
- 现有异步代码（如 `FileMonitor.start`）设计合理，但需注意避免阻塞操作。
- 建议对 CPU 密集型任务（如文件内容分析）使用 `asyncio.to_thread` 或单独进程，防止阻塞事件循环。

### 2.2 资源监控
- `cron_report` 中系统指标采集依赖 `psutil`，但未处理 `psutil` 不可用时的降级方案。
- 可考虑添加内存使用告警，防止内存泄漏。

### 2.3 扩展性
- provider 层的多厂商扩展槽 `PROVIDER_SLOTS` 设计良好，便于添加新 AI 服务。
- 建议为每个 skill 提供标准化的生命周期接口（`start`, `stop`, `status`），便于统一管理。

## 3. 安全性增强

### 3.1 环境变量验证
- 启动时应检查必需的环境变量（如 `DEEPSEEK_API_KEY`、`TELEGRAM_BOT_TOKEN`）是否存在，并提供明确错误提示。

### 3.2 文件系统安全
- `_read_file_preview` 已做后缀白名单，但可进一步限制文件大小（防止读取超大文件）。
- 考虑对监控目录进行符号链接检查，避免循环监控。

### 3.3 网络通信
- 所有外部 API 调用（DeepSeek、Telegram、Ollama）应设置超时和重试机制。
- 建议使用 `httpx` 替代 `urllib`，获得更好的异步支持和连接池管理。

## 4. 后续开发路线图（Phase 2 收尾与 Phase 3 启动）

### 4.1 Phase 2 收尾任务
1. **完善技能集成测试**
   - 编写 `cron_report` 与 `file_monitor` 的集成测试，模拟文件事件和定时任务。
   - 确保 Ollama 心跳检测在 Ollama 服务异常时正确告警。
2. **技能配置化**
   - 将技能参数（如监控目录、告警规则）移至配置文件，支持动态加载。
3. **文档更新**
   - 更新 `README.md`，提供快速启动指南和技能使用示例。
   - 在 `docs/` 下添加技能详细设计文档。

### 4.2 Phase 3 核心任务
1. **Gemini‑CLI 协作激活**
   - 实现 `spawn_gemini_cli()` 方法，允许 Kylopro 调用 Gemini 命令行工具进行复杂任务。
   - 设计 Gemini‑CLI 与现有 provider 层的协作协议（如任务分发、结果回传）。
2. **Docker 容器化部署**
   - 编写 `Dockerfile` 和 `docker-compose.yml`，支持一键容器化部署。
   - 配置容器内日志收集和健康检查。
3. **财务/额度监控模块**
   - 添加 `skills/balance_monitor/`，定期检查 DeepSeek 等 API 的余额和使用情况。
   - 支持阈值告警（如余额低于 10 元时推送 Telegram）。
4. **用户界面增强**
   - 考虑添加简单的 Web 控制面板（使用 FastAPI + Jinja2），用于查看系统状态、管理技能。

### 4.3 长期方向
- **技能市场**：允许用户从 ClawHub 等公共技能库安装第三方技能。
- **多用户支持**：区分不同用户（Telegram chat_id）的配置和技能权限。
- **离线优先**：增强 Ollama 本地处理能力，减少对云端 API 的依赖。

## 6. 风险评估与缓解

| 风险点 | 可能影响 | 缓解措施 |
|--------|----------|----------|
| 第三方 API 变化（DeepSeek、Telegram） | 功能失效 | 封装 API 客户端，定期检查官方文档更新 |
| Ollama 本地模型更新导致接口不兼容 | 文件摘要失败 | 固定 Ollama 模型版本，提供降级方案（不使用摘要） |
| 文件监控占用过高 CPU/内存 | 系统资源紧张 | 增加监控间隔、限制并发处理数，添加资源使用监控 |
| 配置错误导致敏感信息泄露 | 安全风险 | 配置加密存储，启动时验证配置格式，提供配置检查命令 |
